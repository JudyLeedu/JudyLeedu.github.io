最近看了 Claude Code 产品负责人 Cat Wu 的一篇文章，讲 AI 指数级进化下的变化。有些想法想聊聊。

### 边界在消融，但问题不只是"更快"

最直观的感受：AI 把产品团队里各个角色的边界给融了。设计师开始写代码，PM 开始建原型做 eval，工程师开始做产品判断。PM 想到一个 idea，下午就能跑出原型，不用排期等开发。

但"更快"只是表层。

Cat Wu 在文章里点了一个更本质的东西：传统 PM 的底层假设是，项目开始时的技术约束 ≈ 项目结束时的技术约束。所以才有"先调研→写 PRD→交给工程"这个瀑布流式的工作节奏。但模型在指数级进步，你设计时绕过的限制，项目做到一半可能就不存在了。不是快了，是**脚下的地在动**。

### Side Quest：一个下午撞出爆款功能

Side Quest 这个概念我很喜欢。

这个词来自游戏，主线任务之外玩家自己探索的支线。Cat Wu 把它借用到产品里：**roadmap 之外、自驱的、短时间的实验**。不是老板指派的，不是排期规划的，就是你自己觉得"有意思，想试试"，花个下午搞下。

为什么现在 side quest 变得可行？因为 AI 把试错成本压到了接近零。以前验证一个 idea = 写 PRD + 排期 + 开发 + 测试，周期长成本高，所以必须先想清楚再做。现在跟 Claude Code 聊几句，下午就能跑出 demo。**当试错成本趋近于零，"先试再想"就比"先想再试"更合理。**

### PM 的新角色：从规划师到冲浪者

传统 PM 是在确定性里做最优规划，AI 时代的 PM 是在不确定性里加速发现，并识别哪些东西值得坚持。不是控制全流程确保每个细节，而是**识别出少数不可妥协的锚点，其余的放手让团队去试**。

**你不能规划海浪，但你可以学会判断哪道浪值得跟**。

但也有一个困惑，AI 在重塑产品开发的节奏、角色、方法论，但组织设计和管理机制还停在旧范式里。这个裂缝只会越撕越大。也许答案不是"怎么让旧组织适配新节奏"，而是"应该是什么样的新组织形态"。

你们怎么看？
