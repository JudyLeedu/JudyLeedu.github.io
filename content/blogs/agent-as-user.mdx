> Aaron Levie 把 Paul Graham 的名言改写了—"Make something agents want"。听着像段子，但认真想想：当 Agent 比人多几个数量级，"谁在选工具"这件事就变了。

### 万亿 Agent，不是夸张

Levie 的核心判断是：未来会有万亿级 Agent 在跑。写代码、审贷款、处理保险理赔、执行金融交易、当个人助理，你能想到的经济活动，Agent 都会插一手。

万亿这个数听起来大，但逻辑是通的。一个人可以同时跑几十个 Agent，一个公司可以跑几万个。全球每人几个 Agent，量级就上去了。这不是"人被替代"，而是"任务被倍增"。

关键是，量变会带来质变。当 Agent 的数量比人多几个数量级，**"谁在选工具"这件事就变了**。

### 产品经理的焦虑：Agent 不点按钮

之前做产品，用户是活人。做用户访谈，看点击热力图，优化按钮颜色和页面布局。用户觉得好用，就留下来。

但 Agent 不看你的 UI。

Levie 提到："CLIs/APIs are their native tongue"。我们精心打磨的界面，它根本不 care。它只关心：这接口好不好调？文档清不清晰？返回快不快？鉴权麻不麻烦？

换句话说，**API 文档就是 Agent 眼里的"首页"**。

> 这对产品经理来说是个思维切换。以前我们说"用户体验"，默认是人的体验。以后可能得加一层，"Agent 体验"。你的 API 响应慢 200ms，人可能感觉不到，但跑一万个 Agent 的公司会立刻算出成本。

### 高速路还是土路？

Levie 觉得很多基础设施已经有了，Agent 可以直接跑在上面——"the highways have already been built"。我持保留意见。表面上看，身份系统、文件存储、数据库、支付接口，这些都有 API。但**"有 API"和"API 好用"是两回事**。文档缺斤少两、限流标准不透明、鉴权流程复杂、错误码不规范，这些**对人是麻烦，对 Agent 是致命**。Agent 不会"猜一下可能是什么意思"。

所以真正的机会可能不只是"给 Agent 造新工具"，而是**把已有的工具真正 Agent 化**——让 API 从"能用"变成"好用"。

### 谁为 Agent 的错误买单？

Levie 提到了一个很容易被忽视的问题：Agent 不能自己承担责任，谁发起了 Agent，谁就要对结果负责。

但"谁发起"这事没那么简单。是点那个按钮的人？是配置这个 Agent 的团队？还是提供 Agent 平台的厂商？

举个🌰：Agent 自动审批贷款批错了，客户投诉。这个锅谁背？一线操作人员说我只是点了运行，产品经理说我只设计了流程，厂商说模型不是我训练的。灰色地带一大片。

而且 Agent 还有个更麻烦的特性，它不知道自己什么时候跑偏了。Levie 原话是 "they don't quite know when they've run astray"。人会犹豫、会确认、会停下来问一句"这不对吧"，Agent 不会。它只会闷头执行，直到你发现结果不对。

> 围绕 Agent 的"治理"，权限管理、决策可追溯、异常熔断

### 我们是不是该重新学一遍"做产品"了？

Levie 说 "We're entering a completely new era of software development"。我不确定是不是"完全新"，但至少是需要"补课"的。

补啥课？
- 以前做产品，要想人的操作路径。以后得同时想 Agent 的调用路径。
- 以前优化页面，看的是转化率和停留时长。以后可能还得看 API 调用成功率和平均响应时间。
- 以前做开放平台是"加分项"。以后可能是"必选项"。不开放，Agent 就选别人。

这不意味着人不重要了。并不是。而是说，**人和 Agent 同时是你的用户，你得为两种用户都做好体验**。

这事才刚开始，谁也说不准最后会变成什么样。但有一件事是确定的：如果你做产品还在只想着"人怎么用"，可能得开始想想"Agent 怎么调"了。
